
# Отчёт по модулю «Метаболическая карта» для внешнего ревью

Подготовлю единый документ (Markdown + при желании DOCX) с полным описанием сделанного, архитектуры, потоков данных, ограничений и открытых пунктов. Файл сохраню как `docs/metabolic-map/external-review-2026-07.md` (и, по запросу, экспорт в `.docx` в `/mnt/documents/`).

## Структура отчёта

**1. Резюме (1 стр.)**
- Что за модуль, для кого (врач/родитель/исследователь), какую задачу решает.
- Границы ответственности: детерминированные правила vs ИИ-интерпретация vs каталог лечения.

**2. Архитектура**
- Слои: БД (Supabase) → детерминированный агрегатор → Edge-функция ИИ → UI (React/Excalidraw).
- Диаграмма потоков данных: PDF → парсинг → `lab_results`/`patient_diagnosis_timeline` → `metabolic_maps.aggregate_summary` → `map_findings` → `map_recommendations` → печать/родитель/когорта.
- Роли и доступ: `admin`, `editor`, `surgeon`, `parent` (+ `patient_guardians`, флаг `share_simple_only`).

**3. Модель данных**
- Таблицы: `pathways`, `pathway_texts`, `metabolic_maps`, `map_findings`, `map_recommendations`, `metabolic_map_snapshots`, `patient_guardians`, `patient_diagnosis_timeline`, `treatment_catalog` (расширения: `evidence_level`, `catalog_priority`, `age_min/max_years`, `targets`).
- Ключевые поля и связи, RLS-политики, GRANT'ы.

**4. Детерминированный слой**
- `src/lib/metabolic/aggregator.ts`: как правила пути (`rules`) сопоставляются с лабами/антропометрией, как рождается severity (`norm/mild/moderate/severe/no_data/insufficient_data`).
- Набор из 16 путей: список, статус наполненности (11 seeded + placeholder'ы), источники правил.
- `treatmentMatch.ts`: пересечение `targets` каталога с кодами пути/находок, фильтры возраста, противопоказания как предупреждения, ранжирование `priority → evidence_level`.

**5. ИИ-слой (`supabase/functions/metabolic-map-build`)**
- Модель по умолчанию: `google/gemini-2.5-pro` через OpenRouter, `response_format: json_object`, temperature 0.2.
- Что подаётся на вход (draft, справочник путей, лабы, антропометрия, жалобы, анамнез, диагнозы, лекарства, деперсонализация).
- Гардрейлы: не выдумывать значения, ссылка на конкретные показатели, `insufficient_data` при нехватке, потолок severity.
- Куда сохраняется: `metabolic_maps.meta.ai` + AI-`map_findings` (source_ref.ai=true), детерминированные не трогаются.

**6. Графический слой**
- Excalidraw-сцены в `pathways.svg_scene`; авто-layout fallback (`autoLayout.ts`) для пустых.
- Рендер: `PathwaySceneSVG.tsx` → `exportToSvg`, перекраска по `highlights`, окрашивание стрелок по worst severity концов, наложение ℞-маркеров.
- Обзорная «метро»-карта (`MetroOverview.tsx`) и детальные схемы; редактор `PathwayEditor.tsx`.
- Единая легенда `SeverityLegend`, палитра `severityColors.ts`.

**7. UI и печать**
- Кабинет врача: `AdminPatientMetabolicMap` (черновик, ИИ-запуск, редактирование сцен, ℞-блок, dynamics/guardian панели).
- Печать A4: `AdminPatientMetabolicMapPrint` — обзор + лист на затронутый путь с ℞-блоком, только `include_in_print`.
- Родитель: `ParentMetabolicMap` — упрощённый текст без сырых цифр (если `share_simple_only`).
- Плашка в Cabinet: `MetabolicMapMiniCard`.
- Когортная аналитика: `AdminMetabolicCohort` — фильтры, таблицы, CSV-экспорт депер­сонализированных агрегатов.

**8. Динамика во времени**
- `metabolic_map_snapshots` + `DynamicsPanel` — сравнение статусов путей по визитам, хронология диагнозов из PDF (`patient_diagnosis_timeline`), ↑/→/↓ индикаторы.

**9. Безопасность и приватность**
- RLS на всех таблицах; роль `parent` изолирована через `patient_guardians`.
- Деперсонализация в ИИ-запросе (пол/возраст без имени) — опция врача.
- Скрытие сырых цифр для родителя по флагу.

**10. Что не хватает / открытые вопросы (главная секция для ревью)**
- **Клиническая валидация правил**: 11/16 путей имеют реальные пороги, 5 — placeholder; нужен клинический эксперт для докрутки порогов и референсов по возрасту/полу.
- **Валидация ИИ-вывода**: нет автоматического сравнения с эталонными кейсами, нет метрик качества (accuracy/precision по путям), нет audit-trail разночтений «правила vs ИИ».
- **Excalidraw-сцены**: пока авто-layout — визуально бедно; нужны нарисованные врачом сцены для ключевых путей.
- **Каталог лечения**: покрытие `targets` неровное — часть узлов ℞-пустые; нужен аудит соответствия `targets ↔ node_id`.
- **Единицы измерения и синонимы лабов**: зависимость от `lab_synonyms_queue`; при новых лабораториях требуется ручная нормализация.
- **Тесты**: unit-тестов агрегатора и treatmentMatch нет; Edge-функция без интеграционных тестов; нет snapshot-тестов SVG.
- **Производительность когорты**: агрегаты считаются на клиенте; для >1–2k пациентов нужен серверный rollup.
- **Согласия родителей**: нет формализованного flow «врач → приглашение по email → согласие → доступ».
- **i18n**: тексты путей и ℞-блоков только на русском.
- **Логи ИИ-запусков**: нет отдельной таблицы `metabolic_ai_runs` со стоимостью/latency/моделью — только последний снапшот в `meta.ai`.
- **Версионирование правил и сцен**: изменение правила молча меняет историю; нет `rules_version` в снапшотах.

**11. Что попросить у ревьюера**
- Проверку клинических порогов и списка 16 путей.
- Оценку промпта и гардрейлов ИИ.
- Замечания по модели данных (нормализация findings/recommendations).
- Приоритезацию открытых пунктов из раздела 10.
